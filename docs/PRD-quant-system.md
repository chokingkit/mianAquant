# 产品需求文档 (PRD)
## A股量化交易选股系统基于 qlib 框架

---

## 1. 项目信息

**语言**: 中文  
**编程语言**: Python (qlib 框架) + Vite + React + MUI + Tailwind CSS (前端)  
**项目名称**: a_share_quant_system

**原始需求复述**:
基于 qlib 量化框架进行二次开发，构建适配 A 股市场的量化交易选股系统。系统需要支持 A 股股票代码格式、交易日历和市场规则，并采用分层数据源设计，方便用户随时更换数据来源（如从 Tushare 切换到 AKShare）。

---

## 2. 产品定义

### 2.1 产品目标 (Product Goals)

1. **提供 A 股适配的量化研究平台**：在 qlib 基础上扩展 A 股市场特性支持，包括股票代码规范、交易规则、涨跌停限制等
2. **实现灵活的数据源架构**：通过分层设计实现数据源的解耦和快速切换，降低数据迁移成本
3. **降低量化投资门槛**：提供友好的 UI 界面和自动化工作流，让非专业程序员也能进行量化策略研究和回测

### 2.2 用户故事 (User Stories)

1. **As a 量化研究员**, I want 快速切换不同数据源进行策略回测, so that 可以验证数据质量对策略表现的影响

2. **As a 个人投资者**, I want 通过可视化界面运行选股策略, so that 不需要编写代码也能使用量化工具辅助投资决策

3. **As a 量化开发工程师**, I want 在 qlib 框架基础上快速集成新的 A 股数据源, so that 可以灵活应对数据接口变更

4. **As a 投资分析师**, I want 查看策略的历史回测报告和风险评估, so that 可以评估策略的实盘可行性

5. **As a 算法交易员**, I want 将选股策略连接到实盘交易接口, so that 可以实现自动化的交易执行

## 3. 技术规范

### 3.1 需求池 (Requirements Pool)

#### P0 - Must Have (核心功能)

- **R1**: A 股股票代码适配
  - 支持 sh.600000 (上海) 和 sz.000001 (深圳) 格式
  - 自动识别股票代码前缀并路由到对应市场
  
- **R2**: 数据源抽象层设计
  - 定义统一的数据接口 (DataProvider Interface)
  - 实现 Tushare 和 AKShare 两个数据源适配器
  - 支持通过配置文件切换数据源
  
- **R3**: A 股交易日历集成
  - 集成上交所和深交所交易日历
  - 处理节假日、停牌等特殊情况
  
- **R4**: qlib 数据格式转换
  - 将 A 股数据源转换为 qlib 的 Datalog 格式
  - 支持日线、分钟线等多周期数据
  
- **R5**: 基础回测框架
  - 基于 qlib 的回测引擎
  - 支持 A 股交易规则 (T+1, 涨跌停限制)

#### P1 - Should Have (重要功能)

- **R6**: Web UI 界面
  - 策略配置界面
  - 回测结果可视化 (收益曲线、回撤图)
  - 数据源切换配置界面
  
- **R7**: 策略库
  - 预置常用选股策略 (如均线策略、动量策略)
  - 支持自定义策略模板
  
- **R8**: 风险管理模块
  - 仓位控制
  - 止损止盈设置
  - 风险指标计算 (Sharpe Ratio, Max Drawdown)
  
- **R9**: 数据缓存机制
  - 本地缓存减少 API 调用
  - 增量更新支持

#### P2 - Nice to Have (增值功能)

- **R10**: 实盘交易接口
  - 支持券商 API 接入
  - 模拟交易功能
  
- **R11**: 因子研究工具
  - 因子计算和回测
  - 因子有效性分析
  
- **R12**: 多账户管理
  - 支持多个交易账户
  - 绩效对比分析
  
- **R13**: 告警通知
  - 邮件/微信/钉钉通知
  - 策略信号提醒

#### 交互流程

**数据流切换流程**:
1. 用户点击右上角 "数据源: Tushare ▼"
2. 弹出数据源选择菜单 (Tushare / AKShare / 自定义)
3. 选择新数据源后，系统提示需要重新下载数据
4. 用户确认后，后台自动切换并启动数据同步任务
5. 切换完成后提示用户

**策略回测流程**:
1. 用户在策略管理页面选择或创建策略
2. 配置回测参数 (股票池、时间范围、基准)
3. 点击"开始回测"按钮
4. 系统显示进度条，实时展示回测进度
5. 回测完成后自动跳转到结果页面
6. 用户可查看收益曲线、持仓明细、交易记录等

1. **数据源 API 权限**:
   - 用户是否有 Tushare 或 AKShare 的 API Token?
   - 数据更新频率要求 (实时/日频/分钟级)?

2. **部署环境**:
   - 系统是本地部署还是服务器部署?
   - 是否需要支持 Docker 容器化部署?

3. **用户规模**:
   - 预期并发用户数量?
   - 主要是个人使用还是团队协同?

4. **实盘交易需求**:
   - 是否需要连接实盘交易接口?
   - 如果需要，支持哪些券商?

5. **数据源覆盖范围**:
   - 只需要股票数据，还是需要期货、基金、指数等?
   - 需要多长历史数据 (3年/5年/10年)?

6. **性能要求**:
   - 回测速度要求 (例如: 5年日线数据回测在1分钟内完成)?
   - 同时运行多个策略的性能要求?

7. **合规性**:
   - 是否需要审计日志?
   - 数据存储是否符合相关金融监管要求?

## 4. 技术架构建议

### 4.1 数据源分层设计

```
┌─────────────────────────────────────────┐
│       业务逻辑层 (Strategy Engine)       │
├─────────────────────────────────────────┤
│     数据服务层 (Data Service Layer)      │
│  - 统一数据接口定义                      │
│  - 数据缓存管理                         │
│  - 数据质量校验                         │
├─────────────────────────────────────────┤
│   数据源适配层 (Data Adapter Layer)      │
│  - TushareAdapter                       │
│  - AKShareAdapter                       │
│  - CustomAdapter                        │
├─────────────────────────────────────────┤
│     数据源 API (Data Source API)         │
│  - Tushare API                          │
│  - AKShare API                          │
│  - 其他数据源                            │
└─────────────────────────────────────────┘
```

### 4.2 关键技术选型

- **后端框架**: qlib (微软开源量化框架)
- **数据处理**: Pandas, NumPy
- **Web 框架**: FastAPI (后端 API)
- **前端框架**: React + MUI + Tailwind CSS
- **数据库**: 
  - qlib Datalog (时序数据)
  - SQLite/PostgreSQL (业务数据)
- **任务队列**: Celery (异步任务处理)
- **缓存**: Redis (数据缓存)
